# 重新導向

```
建立 '_edirects' 放在跟目錄
```

## 可做功能
功能  說明  狀態碼 範例
1. 永久重新導向 (SEO 友好)  將舊網址永久導向新網址，確保搜索引擎和用戶的書籤不失效，並轉移舊頁面的 SEO 權重。 301 /old-page /new-page 301
2. 暫時重新導向   用於臨時維護或短期的活動網址，不影響搜索引擎的索引。  302 /promo /live-event 302
3. URL 重寫 / SPA Fallback    內部重寫 URL，將請求導向另一個檔案，但 保持瀏覽器地址欄不變。這是您用於 Svelte SPA 的關鍵。  200 /about /index.html 200
4. 自定義 404 頁面   讓所有找不到的資源都導向一個自定義的錯誤頁面，而不是託管服務的預設 404。  404 /* /404.html 404
5. 基於國家/語言的重新導向 根據用戶的地理位置或瀏覽器語言設定將他們導向不同的網址。    302 /cn /zh-tw 302 Language=zh
6. 代理/反向代理 (Proxying)   將您的前端請求透明地轉發到一個外部服務器（例如您的 Supabase Edge Function 或第三方 API），同時隱藏真實的 API 端點。  200 /api/* https://api.external.com/:splat 200
7. 隱藏金鑰 結合環境變數，將敏感資訊隱藏在重新導向規則中，不暴露給前端程式碼。   200 /functions/auth https://<PROJECT_REF>.supabase.co/auth/v1 200



```
# ----------------------------------------------------
# Netlify Redirects Configuration for Svelte SSG Project
# ----------------------------------------------------

# --- 1. Svelte SPA Fallback (Rewrite Rule) ---
# 目的：確保所有不是靜態檔案的路徑 (例如 /about, /contact) 
#       都能正確載入 index.html，讓 Svelte 應用程式接管路由。
# 狀態碼 200 表示這是一個內部重寫 (Rewrite)，而不是外部重新導向 (Redirect)。
/* /index.html    200

# --- 2. 舊網址的永久重新導向範例 (301 Redirect) ---
# 如果產品的 ID 發生變動，舊連結應導向新連結，保持 SEO 權重。
/old-product-id    /product/new-id    301

# --- 3. 攔截和保護範例 (Deny Rule) ---
# 如果發現惡意爬蟲或攻擊，可以透過 IP 或 User Agent 進行攔截。
# /admin-login   /404.html   404!
```
